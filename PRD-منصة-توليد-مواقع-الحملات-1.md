| | |
|---|---|
| **اسم المشروع** | Campaign Site Factory (مصنع مواقع الحملات) |
| **يبني على** | Campaign Management System v4 (النظام الحالي المرفوع) |
| **الإصدار** | 1.0 |

## 1. ملخص تنفيذي

النظام الحالي هو موقع حملة إلكترونية واحد (جهة واحدة) مبني على Vercel + Supabase، يتيح لزوّار الموقع مشاركة "تغريدات" جاهزة على منصة X ضمن حملة هاشتاق موحّدة، مع لوحة إدارة كاملة (حملات، تغريدات، مشرفون فرعيون بصلاحيات، روابط دعوة، تحليلات، تقارير PDF). كل شيء متعلق بالهوية البصرية (الألوان، الشعار، اسم الجهة) وبيانات دخول الأدمن **مُثبّت يدويًا داخل الكود ومتغيرات البيئة لكل نشر منفصل**.

الهدف من هذا المشروع: بناء **"مصنع مواقع" داخلي** يتيح لفريقنا إنشاء نسخة جديدة كاملة من هذا الموقع لأي جهة إعلامية جديدة خلال دقائق، بمجرد إدخال: اسم الجهة، الألوان، الشعار، وبيانات دخول الأدمن — دون لمس كود أو ملفات يدويًا. النظام سينشئ تلقائيًا مشروع Vercel ومشروع Supabase منفصلَين ومعزولَين لكل جهة، باستخدام حساب Vercel/Supabase رئيسي واحد نديره نحن.

بالتوازي، سيتم تحسين المنتج نفسه: إضافة مشاركة عبر واتساب وفيسبوك (وليس X فقط)، رفع مستوى التصميم والانيميشن ليكون احترافيًا بصريًا، وتعزيز الأمان.

## 2. تحليل النظام الحالي (Baseline — من فحص الملفات المرفوعة)

| الملف | الدور | ملاحظات تقنية مهمة |
|---|---|---|
| `index.html` | الصفحة العامة | Tailwind CDN + Alpine.js، الألوان مُعرّفة داخل `tailwind.config` كثابت اسمه `yemen` (green/red...)، عنوان/وصف الصفحة ثابت بالـ `<title>`/`<meta>`، المشاركة حاليًا عبر `x.com/intent/tweet` فقط |
| `admin.html` | لوحة الإدارة | حملات، تغريدات، مشرفون فرعيون بصلاحيات دقيقة (JSONB permissions)، روابط دعوة، تحليلات، تقارير PDF (عبر puppeteer) |
| `____path_.js` (فعليًا `api/[...path].js`) | دالة API واحدة على Vercel | مصادقة، Rate limiting بسيط (IP+UA)، تشفير كلمات مرور المشرفين الفرعيين بـ **PBKDF2 (100000 تكرار) + salt**، جلسات مُخزَّنة كـ **hash فقط** (SHA-256)، مقارنة آمنة زمنيًا (`timingSafeEqual`) |
| `supabase-migration.sql` | مخطط القاعدة | جداول: `sub_admins`, `admin_sessions`, `admin_activity_logs`, `invite_links`, `analytics_events`, بالإضافة لأعمدة على `campaigns`/`tweets` |
| `vercel.json` | إعدادات النشر | Rewrites لـ `/admin` و `/api/*`، رؤوس أمان جيدة (X-Frame-Options, X-Content-Type-Options, Referrer-Policy...) لكن **بدون Content-Security-Policy ولا HSTS** |
| `sw.js` | Service Worker (PWA) | Cache باسم ثابت `campaign-system-v4`، عنوان وأيقونة الإشعارات **مُثبّتة نصيًا** ("لجنة اعتصام أبناء المهرة"، `/logo-dark.png`) |
| `package.json` | الاعتماديات | `@supabase/supabase-js`, `puppeteer-core` + `@sparticuz/chromium` (لتوليد PDF)، Node 18 |

**نقطة الضعف الجوهرية بالنسبة لمشروعنا الجديد:** كل هوية بصرية/نصية (الألوان، الشعار، اسم الجهة، الهاشتاق) **مطبوخة داخل ملفات ثابتة**، وبيانات الأدمن الرئيسي تُقرأ كنص عادي من `ADMIN_USER`/`ADMIN_PASS` (بينما المشرفون الفرعيون مشفّرون بشكل جيد في القاعدة). هذا يجعل تكرار الموقع لجهة جديدة اليوم = نسخ الكود يدويًا + تعديل يدوي في عدة أماكن. هذا بالضبط ما سيحل محله المصنع.

## 3. الرؤية والهدف

> تمكين فريقنا من إطلاق موقع حملة جاهز، آمن، واحترافي بصريًا لأي جهة إعلامية جديدة، من خلال معالج إنشاء بسيط، دون أي تدخل يدوي في الكود أو النشر.

### أهداف قابلة للقياس (Success Metrics)
- إنشاء موقع جديد كامل (نشر + قاعدة بيانات + نطاق) في **أقل من 10 دقائق** من تعبئة المعالج حتى الحصول على رابط حي.
- صفر تعديل يدوي في الكود لإطلاق جهة جديدة.
- كل جهة جديدة معزولة بالكامل (مشروع Vercel + مشروع Supabase خاص بها) — لا تسرّب بيانات بين الجهات بأي شكل.
- دعم 3 منصات مشاركة (X, WhatsApp, Facebook) قابلة للتفعيل/التعطيل لكل جهة.
- عدم كسر أي وظيفة موجودة حاليًا في الموقع الحيّ (الحملات، المشرفون الفرعيون، روابط الدعوة، التحليلات، تقارير PDF).

## 4. نطاق العمل

### داخل النطاق (In Scope)
1. محرك توليد المواقع (Factory Engine) + لوحة تحكم داخلية للمصنع (Super Admin Panel).
2. تحويل القالب الحالي إلى **قالب قابل لإعادة الاستخدام** بهوية بصرية ديناميكية (بدل التثبيت اليدوي).
3. إضافة مشاركة واتساب وفيسبوك (بالإضافة إلى X الموجود).
4. إعادة تصميم احترافي + أنيميشن للموقع العام ولوحة الإدارة.
5. تعزيز الأمان في كل من منصة المصنع والمواقع المُنشأة.
6. إدارة دورة حياة المواقع بعد إنشائها (تعليق، حذف، تحديث الهوية، تدوير كلمة المرور، تحديث القالب لجميع المواقع دفعة واحدة).

### خارج النطاق (Out of Scope حاليًا)
- تسجيل ذاتي للعملاء (Self-service signup) أو نظام دفع/اشتراكات — **قرار مؤكد**: هذه أداة داخلية.
- ربط حسابات Vercel/Supabase الخاصة بالعميل نفسه (OAuth للعميل) — **قرار مؤكد**: نستخدم حسابًا رئيسيًا واحدًا ننشئ تحته مشاريع معزولة.
- دعم لغات غير العربية (i18n) — يبقى النظام عربي RTL كما هو، مع ترك المعمارية مرنة لإضافة لاحقًا إن لزم.
- تطبيقات جوال أصلية (Native apps) — يبقى الحل ويب/PWA كما هو.

## 5. المستخدمون (Personas)

| الدور | من هو | أين يعمل |
|---|---|---|
| **Super Admin (المصنع)** | فريقنا فقط | لوحة تحكم جديدة مركزية "Factory Dashboard" |
| **Main Admin (أدمن الجهة)** | صاحب/مدير الحملة لدى الجهة العميلة | `admin.html` الخاص بموقعه (كما هو حاليًا) |
| **Sub Admin** | مشرف فرعي لدى الجهة | نفس `admin.html`، بصلاحيات محدودة (موجود حاليًا) |
| **زائر الموقع** | الجمهور المشارك في الحملة | `index.html` الخاص بالجهة |

## 6. القرار المعماري (Architecture)

### 6.1 نظامان منفصلان لكن متكاملان

```
┌─────────────────────────────┐        ┌───────────────────────────────────┐
│   Factory Control Plane      │        │   Tenant Site (لكل جهة عميلة)      │
│  (مشروع Vercel + Supabase    │──ينشئ──▶│  مشروع Vercel خاص + مشروع          │
│   واحد، يديره فريقنا فقط)     │        │  Supabase خاص، معزول بالكامل        │
│                              │        │                                     │
│ - Factory Dashboard (UI)     │        │ - نفس كود index.html/admin.html/API │
│ - Provisioning Engine        │        │   (قالب موحّد يُعاد استخدامه)        │
│ - Vercel API client          │        │ - site_settings (هوية ديناميكية)    │
│ - Supabase Mgmt API client   │        │ - نفس مخطط campaigns/tweets/...     │
│ - Secrets Vault (مشفّر)       │        │                                     │
│ - Tenants Registry + Audit   │        │                                     │
└─────────────────────────────┘        └───────────────────────────────────┘
```

### 6.2 القرار التقني الأهم: الهوية البصرية "ديناميكية" وليست "مطبوخة بالكود"

بدل تعديل ملفات HTML/Tailwind لكل جهة (نهج هش وصعب الصيانة)، **الكود مصدر واحد يُنشر كما هو لكل الجهات**، والاختلاف الوحيد بين المواقع هو:
1. متغيرات بيئة الاتصال بقاعدة بيانات تلك الجهة فقط (`SUPABASE_URL`, `SUPABASE_KEY`).
2. صفّ إعدادات واحد في جدول جديد `site_settings` داخل قاعدة بيانات الجهة نفسها (اسم الجهة، الشعار، الألوان، الهاشتاق، المنصات المفعّلة...).

الصفحات (`index.html`, `admin.html`) تُقرأ عبر دالة عرض من جانب الخادم (Server-Side Rendering بسيط داخل دالة Vercel) تجلب `site_settings` وتُدرجها في:
- `<title>` و `<meta description>` (لأغراض SEO ومنع "وميض" هوية خاطئة).
- سكربت `window.__SITE_CONFIG__` يُحقن قبل تحميل Tailwind، ليقرأه `tailwind.config` بدل الألوان الثابتة.
- `sw.js` يصبح ديناميكيًا هو الآخر (اسم الجهة والشعار في الإشعارات تُجلب عبر `fetch('/api/config')` وقت حدث `push` بدل النص الثابت).

**لماذا هذا القرار؟** لأنه يجعل تحديث القالب (تصميم جديد، إصلاح أمني، ميزة جديدة) عملية "إعادة نشر نفس الكود لكل المشاريع" بدل "تعديل يدوي متكرر" — وهذا شرط أساسي لتحقيق هدف "التوليد بلا لمس كود".

### 6.3 لماذا مشروع Supabase منفصل لكل جهة (وليس تعدد مستأجرين على قاعدة واحدة)
بناءً على قرارك، كل جهة تحصل على مشروع Supabase معزول بالكامل. هذا يعطي:
- عزل بيانات فعلي 100% (لا حاجة لمنطق "فلترة حسب tenant_id" عرضة للأخطاء).
- إمكانية تعليق/حذف جهة بالكامل بأمان دون التأثير على غيرها.

⚠️ **مخاطرة يجب رصدها**: خطط Supabase المجانية/المدفوعة لها **حد أقصى لعدد المشاريع لكل مؤسسة (Organization)**، ولكل مشروع تكلفة/حصة منفصلة. يجب على فريقنا التحقق من خطة Supabase Organization الحالية وتوسيعها حسب عدد الجهات المتوقع (مذكور في قسم المخاطر).

## 7. تدفق إنشاء موقع جديد (End-to-End Flow)

### أ. معالج الإنشاء (Wizard) في لوحة المصنع
1. **بيانات الجهة**: اسم الجهة، وصف مختصر، الهاشتاق الافتراضي، رفع الشعار (وأيقونة favicon).
2. **الهوية البصرية**: اللون الأساسي، اللون الثانوي/التنبيهي، وضع الواجهة الافتراضي (داكن/فاتح)، معاينة حيّة (Live Preview) قبل الإنشاء.
3. **بيانات دخول الأدمن الرئيسي**: اسم مستخدم + كلمة مرور (مع مؤشر قوة كلمة المرور وزر "توليد كلمة مرور قوية").
4. **منصات المشاركة**: تفعيل/تعطيل X، واتساب، فيسبوك (افتراضيًا الثلاثة مفعّلة).
5. **النطاق**: نطاق فرعي تلقائي تحت نطاقنا (مثال: `orgname.campaigns.ourdomain.com`) أو خيار "نطاق مخصص لاحقًا" (تعليمات DNS تُعرض للجهة).
6. **مراجعة وتأكيد** → زر "إنشاء الموقع الآن".

### ب. خط أنابيب التزويد التلقائي (Provisioning Pipeline) — تشغيل غير متزامن (Async Job)
1. إنشاء مشروع Supabase جديد عبر Supabase Management API.
2. تشغيل ملف الهجرة (migration SQL) تلقائيًا على المشروع الجديد + إضافة جدول `site_settings` الجديد.
3. إنشاء Storage bucket ورفع الشعار/الأيقونة إليه.
4. إدخال صف `site_settings` بالبيانات المُدخلة في المعالج.
5. إنشاء مستخدم Admin رئيسي داخل قاعدة الجهة (كلمة مرور مُشفّرة PBKDF2/Argon2 — **وليس متغير بيئة نصي**).
6. إنشاء مشروع Vercel جديد عبر Vercel API من نفس قالب الكود (Git repo واحد مشترك).
7. ضبط متغيرات البيئة على مشروع Vercel الجديد (`SUPABASE_URL`, `SUPABASE_SERVICE_KEY` فقط — لا كلمات مرور نصية).
8. تشغيل النشر (Deployment) ومتابعة حالته.
9. ربط النطاق الفرعي عبر Vercel Domains API.
10. فحص صحي (Health check) للموقع الجديد + `/admin`.
11. تسجيل النتيجة في `tenants` + عرض رابط الموقع ورابط الإدارة لمرة واحدة فقط في لوحة المصنع (ثم لا تُعرض كلمة المرور مجددًا — الجهة تُبلَّغ لتغييرها عند أول دخول).
12. عند فشل أي خطوة: إيقاف العملية، تسجيل الخطأ، **حذف أي موارد تم إنشاؤها جزئيًا** (Vercel project / Supabase project) لتفادي موارد يتيمة وتكاليف غير ضرورية، وإتاحة "إعادة المحاولة" بضغطة زر.

## 8. المتطلبات الوظيفية التفصيلية

### FR1 — لوحة تحكم المصنع (Factory Dashboard)
- تسجيل دخول خاص بفريقنا فقط، مع **مصادقة ثنائية (2FA)** إلزامية (نظرًا لحساسية الصلاحيات: توكنات تصل لكل بنية تحتية العملاء).
- عرض قائمة كل الجهات: الاسم، الحالة (قيد الإنشاء/نشطة/معلّقة/فشل)، رابط الموقع، تاريخ الإنشاء.
- إجراءات لكل جهة: فتح الموقع، فتح لوحة إدارتها، تعليق، حذف (مع تأكيد صريح مزدوج)، إعادة تعيين كلمة مرور الأدمن الرئيسي، تعديل الهوية البصرية بعد الإنشاء.
- سجل تدقيق (Audit Log) لكل إجراء يتم على أي جهة، مع اسم المُنفّذ والوقت وIP.

### FR2 — معالج إنشاء موقع جديد
- كما في القسم 7-أ بالضبط، مع تحقق فوري من المدخلات (تفرد اسم الجهة/النطاق الفرعي، قوة كلمة المرور، صيغة/حجم الشعار).
- معاينة حيّة للألوان والشعار قبل التأكيد.

### FR3 — محرك التزويد (Provisioning Engine)
- تنفيذ الخطوات في القسم 7-ب كوظيفة غير متزامنة (Job Queue) مع شريط تقدّم حيّ في الواجهة (WebSocket أو Polling).
- معالجة أخطاء شاملة + Rollback تلقائي عند الفشل.
- إمكانية "تحديث القالب لجهة موجودة" (إعادة نشر نفس الكود بأحدث نسخة) وأيضًا **"تحديث جماعي لكل الجهات"** بضغطة واحدة من لوحة المصنع.

### FR4 — نظام الهوية البصرية الديناميكية (site_settings)
- جدول `site_settings` (تفصيل في القسم 9) يقرأه كل من: صفحة العرض العامة، لوحة الإدارة، و`sw.js`.
- نقطة API عامة للقراءة فقط: `GET /api/config` (بدون مصادقة، بيانات غير حساسة فقط: الاسم، الشعار، الألوان، الهاشتاق، المنصات المفعّلة).
- تحديث الهوية من لوحة إدارة الجهة نفسها (اختياري) أو من لوحة المصنع فقط (يُحسم لاحقًا — افتراضيًا: من لوحة المصنع، مع إمكانية فتحها لاحقًا لأدمن الجهة).

### FR5 — مشاركة متعددة المنصات (X + WhatsApp + Facebook)
- توسيع دالة المشاركة الحالية (`x.com/intent/tweet`) لتشمل:
  - **واتساب**: `https://wa.me/?text=<نص مُرمّز>`
  - **فيسبوك**: `https://www.facebook.com/sharer/sharer.php?u=<رابط مُرمّز>` (ملاحظة تقنية: فيسبوك لا يضمن عرض نص مخصص عبر باراميتر `quote` دائمًا — يُعتمد أساسًا على معلومات Open Graph من الرابط نفسه، لذا يجب التأكد من وسوم `og:title`/`og:description`/`og:image` صحيحة وديناميكية لكل جهة).
  - **X**: كما هو حاليًا.
- كل زر مشاركة يُظهر فقط إن كانت المنصة مفعّلة في `site_settings.enabled_share_platforms`.
- تتبّع كل نقرة مشاركة حسب المنصة في `analytics_events` (إضافة عمود/قيمة `platform`) لإثراء لوحة التحليلات الحالية بمقارنة أداء كل منصة.
- تصميم أيقونات/أزرار مطابقة للهوية البصرية لكل منصة (ألوان العلامات التجارية الرسمية) ضمن نفس منطق الأنيميشن العام.

### FR6 — إعادة تصميم احترافي + أنيميشن
- مراجعة شاملة لنظام التصميم الحالي (Glassmorphism الموجود) مع: تدرّج طباعي أوضح، مسافات متّسقة، حالات تحميل (Skeleton loaders) بدل الفراغ، عدّاد تصاعدي متحرك لعدد المشاركات، انتقالات صفحة سلسة، تأثيرات دقيقة (micro-interactions) عند الضغط على أزرار المشاركة.
- تطبيق التحسين **مرة واحدة على القالب المشترك** فقط — يستفيد منه كل موقع حالي ومستقبلي تلقائيًا عبر آلية "تحديث القالب" في FR3.
- الحفاظ على الوضعين الداكن/الفاتح والدعم الكامل لـ RTL كما هو حاليًا.

### FR7 — تعزيز الأمان
تفصيل كامل في القسم 11.

### FR8 — إدارة دورة حياة المواقع
- تعليق موقع (يوقف الوصول العام مؤقتًا دون حذف البيانات).
- حذف موقع نهائيًا (حذف مشروع Vercel + مشروع Supabase فعليًا، بعد تأكيد مزدوج وفترة سماح Grace Period اختيارية).
- تدوير/إعادة تعيين بيانات دخول الأدمن الرئيسي لأي جهة من لوحة المصنع مباشرة.

## 10. التكاملات الخارجية المطلوبة

| التكامل | الاستخدام | ملاحظات |
|---|---|---|
| **Vercel REST API** | إنشاء مشروع، ضبط متغيرات بيئة، تشغيل نشر، ربط نطاق، حذف مشروع | يتطلب Vercel API Token بصلاحيات فريق/مؤسسة (Team scope)، يُخزَّن في `secrets_vault` فقط |

## 11. متطلبات الأمان التفصيلية

> ملاحظة مهمة على الترتيب: طلبتَ أن يكون تعزيز الأمان أولوية رابعة ضمن خارطة الطريق العامة — وهذا محترم في الجدولة أدناه. لكن بعض بنود الأمان هنا **أساسية وغير قابلة للتأجيل داخل المرحلة الثانية نفسها (بناء المحرك)**، لأن هذا المحرك سيحمل مفاتيح وصول لبنية تحتية *كل* الجهات دفعة واحدة — أي اختراق له يعني اختراق الجميع، وهذا أخطر بكثير من اختراق موقع جهة واحدة.

### بنود أساسية (يجب أن تُبنى ضمن المرحلة 2 نفسها):
1. تشفير توكنات Vercel/Supabase الرئيسية في `secrets_vault` (AES-256-GCM كحد أدنى)، ولا تُعرض هذه التوكنات أبدًا في أي واجهة أو سجل (log).
2. صلاحيات محدودة قدر الإمكان لتوكن Vercel/Supabase (Least Privilege) بدل توكن مالك كامل إن أمكن.
3. مصادقة ثنائية (2FA) إلزامية لحسابات `super_admins`.
4. نقل الأدمن الرئيسي لكل جهة من متغير بيئة نصي إلى تخزين مشفّر PBKDF2/Argon2id داخل قاعدة الجهة (نفس مستوى حماية المشرفين الفرعيين حاليًا أو أفضل).
5. سجل تدقيق كامل (`factory_activity_logs`) لا يمكن حذفه أو تعديله من الواجهة.

### بنود ضمن مرحلة تعزيز الأمان الشاملة (المرحلة 4):
6. إضافة **Content-Security-Policy** و **HSTS** لرؤوس `vercel.json` (غير موجودة حاليًا).
7. قفل حساب مؤقت بعد عدد محاولات دخول فاشلة لكل حساب (Account Lockout) — إضافة فوق الحماية الحالية المعتمدة على IP فقط.
8. إجبار تغيير كلمة المرور عند أول دخول لأي أدمن رئيسي جديد يُنشأ عبر المصنع.
9. ترقية اختيارية لخوارزمية التجزئة من PBKDF2 إلى Argon2id لزيادة المقاومة لهجمات GPU.
10. مراجعة أمنية دورية (Dependency scanning) للحزم المستخدمة (`puppeteer-core`, `@supabase/supabase-js`...).
11. تنبيه فوري (Alert) لفريقنا عند أي فشل متكرر في محاولات الدخول للمصنع نفسه أو عند أي عملية حذف/تعليق لجهة.

## 12. متطلبات غير وظيفية

- **الأداء**: إنشاء موقع جديد كامل ≤ 10 دقائق من التأكيد حتى الرابط الحي.
- **الموثوقية**: كل خطوة في خط الأنابيب قابلة لإعادة المحاولة (Idempotent) دون تكرار موارد مكررة.
- **قابلية الرصد (Observability)**: لوحة حالة لكل عملية إنشاء (نجاح/فشل/قيد التنفيذ) + سجلات مفصّلة للأخطاء.
- **التكلفة**: يجب توثيق التكلفة التقديرية لكل جهة جديدة (مشروع Vercel + مشروع Supabase) والتأكد من خطة Supabase Organization تسمح بالعدد المتوقع من الجهات.
- **التوافقية العكسية**: الموقع الحيّ الحالي يجب ألا يتأثر سلبًا أثناء التطوير — يُنصح بمعاملته كـ "الجهة رقم 1" وتهجيره تدريجيًا لنفس القالب الجديد بعد اختبار المصنع على جهة تجريبية أولاً.

## 13. خارطة الطريق والمراحل (مبنية على أولوياتك)

### المرحلة 1 — مشاركة واتساب وفيسبوك (الأسرع تنفيذًا، أعلى أولوية)
- تعديل منطق المشاركة في `index.html`/API الحالي مباشرة (على الموقع الحيّ وفي القالب المستقبلي معًا).
- **معيار الإنجاز**: زائر يمكنه المشاركة عبر X، واتساب، وفيسبوك، مع تتبّع كل نقرة حسب المنصة في التحليلات.

### المرحلة 2 — محرك التوليد (Factory Engine)
- بناء قاعدة بيانات المصنع، لوحة تحكم المصنع، معالج الإنشاء، خط أنابيب التزويد (Vercel API + Supabase Management API)، وتحويل القالب لهوية ديناميكية (`site_settings` + `/api/config`).
- يتضمن البنود الأمنية "الأساسية" المذكورة في القسم 11 كجزء لا يتجزأ من هذه المرحلة.
- **معيار الإنجاز**: إنشاء جهة تجريبية كاملة من الصفر عبر المعالج فقط، دون أي لمسة كود يدوية، وتعمل بكل ميزاتها الحالية (حملات، مشرفون، روابط دعوة، تحليلات، تقارير PDF).

### المرحلة 3 — إعادة التصميم الاحترافي + الأنيميشن
- تطبَّق مرة واحدة على القالب المشترك، ثم تُدفع لكل الجهات القائمة عبر "تحديث جماعي للقالب" (ميزة من FR3).

### المرحلة 4 — تعزيز الأمان الشامل
- تنفيذ البنود المتبقية في القسم 11 (CSP، HSTS، Account lockout، إجبار تغيير كلمة المرور، ترقية التجزئة، فحص الاعتماديات).

## 14. المخاطر والافتراضات

| الخطر | الأثر | التخفيف |
|---|---|---|
| حد عدد مشاريع Supabase في خطة المؤسسة الحالية | لن نستطيع إنشاء جهات جديدة بعد الوصول للحد | التحقق من الخطة الحالية ورفعها مبكرًا قبل إطلاق المصنع |
| تكلفة تراكمية لكل مشروع Vercel/Supabase جديد | ارتفاع الفاتورة الشهرية مع نمو عدد الجهات | تقدير تكلفة لكل جهة وتضمينه في نموذج تسعير داخلي مستقبلي |
| فشل جزئي أثناء التزويد (موارد يتيمة) | مشاريع منشأة جزئيًا تستهلك حصة/تكلفة دون فائدة | آلية Rollback تلقائي إلزامية (مذكورة في القسم 7-ب) |
| رسالة فيسبوك المخصّصة قد لا تظهر دائمًا | تجربة مشاركة غير متسقة على فيسبوك | الاعتماد على وسوم Open Graph الديناميكية بدل الاعتماد فقط على باراميتر النص |
| توكن Vercel/Supabase الرئيسي نقطة فشل واحدة (Single Point of Failure) | اختراقه = خطر على كل الجهات دفعة واحدة | تشفير + صلاحيات محدودة + 2FA + سجل تدقيق (القسم 11) |

## 15. قرارات مؤجلة تحتاج تأكيدك لاحقًا
1. هل تحديث الهوية البصرية بعد الإنشاء يتم فقط من لوحة المصنع، أم يُسمح لأدمن الجهة نفسه بتغيير شعاره/ألوانه من لوحته؟
2. استراتيجية النطاقات: نطاق فرعي تحت نطاقنا فقط، أم دعم نطاقات مخصصة لكل جهة منذ الإصدار الأول؟
3. هل نحتاج فترة سماح (Grace Period) قبل الحذف النهائي لأي جهة مُلغاة؟

## 16. تعليمات مباشرة للذكاء الاصطناعي المنفّذ

- **لا تُعدّل أو تحذف** أي وظيفة موجودة حاليًا (الحملات، التغريدات، المشرفون الفرعيون، روابط الدعوة، التحليلات، تقارير PDF) — الهدف "إضافة" و"تعميم"، وليس إعادة الكتابة من الصفر.
- ابدأ دائمًا بالمرحلة 1 (مشاركة واتساب/فيسبوك) لأنها أقل خطرًا وأسرع قيمة، ثم انتقل للمرحلة 2.
- عند بناء المرحلة 2، اختبر خط أنابيب التزويد الكامل على **جهة تجريبية وهمية واحدة** أولًا قبل ربطه بأي بيانات إنتاج حقيقية.
- أي تعامل مع توكنات Vercel/Supabase الرئيسية يجب أن يمر فقط عبر الخادم (Server-side)، ولا يظهر مطلقًا في كود الواجهة الأمامية أو في أي استجابة API للعميل.
- حافظ على نمط التشفير المستخدم حاليًا (PBKDF2 + salt + مقارنة زمنية آمنة) كحد أدنى عند إنشاء الأدمن الرئيسي الجديد لكل جهة، وطبّقه بدل الاعتماد على متغيرات بيئة نصية.
- كل ميزة جديدة (مشاركة، تصميم) يجب أن تُبنى في **القالب المشترك فقط**، بحيث تُورَّث تلقائيًا لكل جهة عبر آلية تحديث القالب، لا أن تُكتب لكل جهة على حدة.

## 17. معايير النجاح الكلية للمشروع
- إنشاء جهة إعلامية جديدة بالكامل خلال دقائق، بدون أي تدخل كود يدوي.
- 3 منصات مشاركة تعمل بشكل صحيح ومتتبَّعة إحصائيًا.
- تصميم احترافي موحّد يُطبَّق تلقائيًا على كل المواقع الحالية والمستقبلية.
- عزل بيانات كامل بين الجهات، مع أمان معزَّز لطبقة المصنع نفسها (2FA + تشفير التوكنات + سجل تدقيق).
